# 番茄工作钟 macOS 前台应用追踪版设计

日期：2026-07-03

## 1. 目标

在现有 Tauri + React + TypeScript 项目中开发一套三页番茄工作钟桌面应用，视觉上对齐当前 `.ardot` 设计稿，功能上支持：

- 主计时页：专注 / 短休息 / 长休息模式切换，开始 / 暂停 / 重置，真实倒计时，进度展示。
- 统计页：当天专注总时长、完成番茄数、历史 session 列表。
- 详情页：单次专注 session 的起止时间、总时长，以及 session 期间前台应用使用占比。
- 原生追踪：仅支持 macOS，仅记录前台应用名与 `bundle id`，不记录窗口标题、不记录输入内容、不记录站点或文件路径。
- 本地持久化：完成的专注 session 与应用分布结果落到本地，应用重启后可恢复统计与详情。

这是一个“页面 + 原生采集 + 本地数据”一体化项目，不是单纯静态页面还原。

## 2. 已确认边界

### 2.1 平台边界

- 仅支持 `macOS`。
- 不做 Windows 兼容层。
- 不做 Linux 兼容层。

### 2.2 追踪边界

- 仅在“专注模式运行中”采样前台应用。
- 仅记录：
  - `appName`
  - `bundleId`
  - 采样时间点或采样区间
- 不记录：
  - 窗口标题
  - 文档名
  - 浏览器标签页
  - 键盘输入
  - 鼠标输入
  - 后台进程

### 2.3 交付边界

- 做完整三页流程。
- 做真实本地数据，而不是 mock 数据。
- 不做菜单栏常驻。
- 不做系统通知。
- 不做跨天、跨周、跨月复杂报表。
- 不做权限引导页的高级包装，只做必要错误提示。
- 不做后台 daemon。

## 3. 视觉与交互方向

界面应尽量贴合当前 `.ardot` 设计稿：

- 奶油色背景作为主底色。
- 黑色粗边框与强对比排版。
- 品牌强调色使用亮红与亮黄。
- 主页面保留大数字倒计时、粗框按钮、标签式模式切换、几何装饰块。
- 统计页与详情页保持纸卡片 / 清单式布局，不引入额外花哨组件。

允许的偏离只有一处：主页面新增一条非常轻量的“当前记录应用”信息，例如：

`当前记录：VS Code`

这条信息用于让真实追踪能力可见，避免用户无法判断当前统计来源。它应放在不破坏主视觉结构的位置，例如按钮区与模式切换区之间，或查看统计按钮上方。

## 4. 核心使用流程

### 4.1 主计时流程

1. 用户打开应用，默认进入主计时页。
2. 默认模式为“专注”，显示默认时长，例如 `25:00`。
3. 用户点击开始后：
   - 倒计时启动。
   - Rust 侧创建“进行中的专注 session”。
   - 原生前台应用采样器启动。
4. 计时过程中前端每秒更新剩余时间，原生侧按固定频率采样前台应用。
5. 用户点击暂停后：
   - 倒计时暂停。
   - 原生采样暂停。
6. 用户点击继续后：
   - 倒计时继续。
   - 原生采样继续。
7. 用户点击重置后：
   - 倒计时恢复到当前模式默认时长。
   - 未完成 session 被丢弃。
   - 本次采样数据不进入历史。
8. 当专注倒计时自然结束后：
   - 原生侧结束 session。
   - 聚合应用分布。
   - 写入本地历史。
   - 更新当天统计。

### 4.2 统计页流程

1. 用户在主页面点击“查看统计”进入统计页。
2. 页面展示当天聚合结果：
   - 今日专注总时长
   - 完成番茄数量
   - 历史 session 列表
3. 点击某条历史，进入详情页。

### 4.3 详情页流程

1. 详情页基于选中的单条 session 加载。
2. 展示：
   - 起止时段
   - 总时长
   - 各应用耗时
   - 各应用占比
3. 点击返回，回到统计页。

## 5. 架构决策

采用“前端主导，Rust 负责采集与存储”的结构。

### 5.1 React 负责

- 页面路由 / 三页切换
- 倒计时 UI 状态
- 模式切换 UI
- 统计页与详情页展示
- 对 Tauri commands 的调用
- 运行态错误提示

### 5.2 Rust 负责

- macOS 前台应用信息读取
- session 生命周期持久化
- 采样数据累积与聚合
- 统计汇总查询
- 详情查询
- 本地文件存储读写

### 5.3 不采用的结构

- 不把完整业务状态都塞到 Rust。当前项目是前端空模板，这会平白增加调试与开发成本。
- 不把采样后的历史数据只存 WebView 本地存储。那会造成“原生采样，前端存储”的断层，窗口关闭恢复与异常中断处理都会变脆。

## 6. 前端结构设计

建议前端至少拆成以下模块：

- `AppShell`：三页容器与全局数据加载。
- `TimerPage`：主计时页。
- `StatsPage`：统计页。
- `SessionDetailPage`：详情页。
- `usePomodoroTimer`：纯前端倒计时逻辑与模式状态。
- `usePomodoroData`：与 Tauri commands 交互，刷新统计、详情与历史。
- `components/`：按钮、模式标签、统计卡片、历史项、详情行等视觉复用单元。

页面切换无需上重量级路由。当前只有三页，使用一个轻量级的页面状态机即可：

- `timer`
- `stats`
- `detail`

详情页额外携带 `selectedSessionId`。

## 7. 原生采样设计

### 7.1 采样时机

只在“专注模式且正在运行”时采样。

以下情况不采样：

- 短休息
- 长休息
- 暂停
- 重置后待机
- 应用尚未开始专注

### 7.2 采样内容

每次采样获取当前前台应用：

- `bundleId`
- `localizedName/appName`
- `sampledAt`

### 7.3 采样频率

建议固定频率轮询，首版使用 `1 秒` 采样一次。

理由：

- 实现简单、确定性强。
- 对番茄钟这类分钟级统计足够。
- 后续若需要优化，可替换为事件驱动或混合模式。

### 7.4 聚合策略

专注 session 完成后，将采样点按应用聚合为总秒数。

规则：

- 相邻采样点默认按一个采样间隔记账。
- 最终结果按应用聚合为 `durationSeconds`。
- 再计算 `percentage = durationSeconds / sessionDurationSeconds`。
- 最终详情页按使用时长降序展示。

### 7.5 异常情况

- 如果采样时拿不到前台应用：
  - 记为 `Unknown` 或跳过该秒。
  - 但不能导致整个 session 失败。
- 如果 session 完成时没有任何有效采样：
  - session 仍可写入历史。
  - 详情页显示空分布或“未采集到应用数据”。

## 8. 数据模型

建议 Rust 侧持久化以下结构。

### 8.1 TimerSettings

- `focusDurationSeconds`
- `shortBreakDurationSeconds`
- `longBreakDurationSeconds`
- `longBreakInterval`

首版可以写死默认值，不必须开放设置 UI，但结构上应保留。

### 8.2 SessionRecord

- `id`
- `mode`，仅持久化完成的 `focus`
- `startedAt`
- `endedAt`
- `durationSeconds`
- `completed`，首版完成 session 固定为 `true`

### 8.3 SessionAppUsage

- `sessionId`
- `bundleId`
- `appName`
- `durationSeconds`
- `percentage`

### 8.4 RuntimeSessionState

仅内存态，不必直接持久化：

- 当前模式
- 是否运行中
- 剩余秒数
- 当前进行中的 session id 或内存句柄
- 当前已收集采样缓存

## 9. 本地存储方案

Rust 侧将数据写入应用本地目录中的 JSON 文件或轻量数据库。首版优先使用 JSON 文件，理由是实现更快、结构透明、便于调试。

建议至少拆为：

- `sessions.json`
- `app_usage.json`
- 可选 `settings.json`

如果实现过程中发现查询拼装过于笨重，再切 SQLite；但首版 spec 以 JSON 持久化为默认。

## 10. Tauri Command API 草案

前端与 Rust 之间至少需要这些 commands：

- `get_app_bootstrap_state`
  - 返回默认时长、今日统计、最近历史。
- `start_focus_session`
  - 开始一次专注 session，并启动采样。
- `pause_focus_session`
  - 暂停当前 session 的计时与采样。
- `resume_focus_session`
  - 恢复当前 session 的计时与采样。
- `reset_focus_session`
  - 放弃当前未完成 session。
- `finish_focus_session`
  - 完成当前 session，写入历史并返回最新统计。
- `get_today_stats`
  - 获取今日专注总时长与完成数。
- `list_sessions`
  - 获取历史 session 列表。
- `get_session_detail`
  - 获取某条 session 的详情与应用分布。
- `get_current_foreground_app`
  - 可选，仅用于主页面显示“当前记录应用”。

是否暴露 `get_current_foreground_app` 可以在实现时决定：

- 如果原生采样器内部已经维护了“最近一次前台应用”，前端可以通过状态查询拿到。
- 不需要为了 UI 显示而重复做两套采集。

## 11. 页面行为细则

### 11.1 主计时页

- 模式切换仅在未运行时允许，或切换时先强制重置。
- 开始按钮在运行中变为禁用或隐藏。
- 暂停按钮仅在运行中可用。
- 重置按钮在已开始后可用。
- 完成数展示当天已完成的专注 session 数量。
- 进度条按当前模式总时长计算。
- “当前记录应用”只在专注运行中显示真实值；其余状态显示：
  - `未开始`
  - `已暂停`
  - 或直接隐藏

### 11.2 统计页

- 顶部两张卡分别显示：
  - 今日专注总时长
  - 完成番茄数
- 历史列表按时间倒序排列。
- 列表项至少展示：
  - 日期
  - 起止时间
  - 模式标签
  - 时长

### 11.3 详情页

- 标题显示“使用详情”。
- 顶部黑底信息条显示起止时间和总时长。
- 应用列表按耗时降序排列。
- 每行展示：
  - 应用色块或占位图形
  - 应用名
  - 时长
  - 占比
  - 横向进度条

## 12. 错误处理

### 12.1 无法读取前台应用

- 主页面应提示“无法读取前台应用，统计可能不完整”。
- 但倒计时本身不应被阻塞。

### 12.2 本地读写失败

- 首屏加载失败时展示可重试错误态。
- session 完成落盘失败时应明确告知用户“本次统计未保存成功”。

### 12.3 数据为空

- 历史为空时显示空状态，而不是假数据。
- 详情分布为空时显示空态说明。

### 12.4 macOS 权限或系统限制

- 如果当前实现依赖的系统 API 在用户环境下失败，前端要显示明确信息。
- 首版只要求错误可见，不要求完整权限引导流程。

## 13. 测试策略

### 13.1 前端测试

- 倒计时状态机：
  - 开始
  - 暂停
  - 恢复
  - 重置
  - 模式切换
- 统计页数据映射。
- 详情页排序与百分比展示。

### 13.2 Rust 测试

- 采样点聚合为应用耗时的纯函数测试。
- session 完成写入后的统计汇总测试。
- 历史列表与详情查询测试。

### 13.3 手工验证

- 主页面视觉是否贴稿。
- 专注完成后统计页是否立即更新。
- 点击历史进入详情是否正确。
- 同一次 session 中切换不同前台应用，详情占比是否变化。

## 14. 非目标

以下内容明确不在本次实现范围内：

- 自动识别网站、文件、项目路径
- 跨设备同步
- 用户账户
- 云端存储
- 通知中心集成
- 菜单栏模式
- 深色模式
- 系统级后台守护进程
- 多语言

## 15. 实施顺序建议

1. 先完成三页 React 结构与静态贴稿。
2. 再接入前端倒计时状态机。
3. 再在 Rust 中实现本地 session 存储与查询。
4. 再接入 macOS 前台应用采样。
5. 最后完成统计、详情联通与手工验证。

这个顺序能保证页面先成形，原生能力后接入，不会在默认模板上同时把视觉、状态、原生、存储四件事搅在一起。
